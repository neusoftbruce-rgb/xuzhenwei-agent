# 徐振伟智能体 v2.4 要件定义书 — 咨询式分析

## 1. 问题诊断

### 1.1 流中断根因
- DeepSeek API `max-tokens: 4096` 默认值，复杂问题回复超过4096 tokens即被截断
- 前端无"继续生成"机制

### 1.2 Token浪费根因
- 深度思考模式每次2次API调用（reasoner 8000 tokens + chat polish）
- 70%的简单问题不需要深度思考，但用户默认开启
- 系统提示词过长（AgentEngine AGENT_SYSTEM_PROMPT ~ 400 tokens）
- 历史对话全量注入，随对话增长Token线性增长

### 1.3 体验问题
- 无"快速概览→选择深挖"的渐进式分析流程
- 缺少Token消耗透明度

## 2. 解决方案: 咨询式两阶段分析

### 2.1 核心思路
参考 McKinsey Pyramid Principle + MECE 框架：
1. **全景快扫**（Quick Scan, ~1500 tokens）
2. **按需深挖**（Deep Dive, ~2000 tokens/维度）
3. **综合建议**（Synthesis, ~1000 tokens）

### 2.2 新增功能

| ID | 功能 | 说明 |
|----|------|------|
| CS-01 | 咨询模式入口 | 侧边栏新增 "🏛️ 咨询式分析" 模式 |
| CS-02 | 全景快扫 | 快速模型生成结构化MECE分析，3-5个维度，每维度1-2句话 |
| CS-03 | 可点击深挖 | 每个维度附带"🔍 深挖"按钮，点击触发深度分析 |
| CS-04 | Token仪表盘 | 显示本次会话Token消耗（估算） |
| CS-05 | 继续生成 | 检测截断并自动续接 |
| CS-06 | 提示词优化 | 精简系统提示词，添加输出长度控制 |

### 2.3 Token优化

| 优化项 | 节省 | 实现方式 |
|--------|------|----------|
| 快扫用快模型 | ~70% | consulting endpoint 用 deepseek-chat + low max_tokens |
| 系统提示词压缩 | ~30% | 从400 tokens压缩到150 tokens |
| 历史裁剪 | ~50% | 只注入最近3轮，超长内容摘要化 |
| 按需深挖 | ~60% | 用户选择后才调用reasoner |
| Token仪表盘 | N/A | 前端展示估算消耗，用户自主控制 |
