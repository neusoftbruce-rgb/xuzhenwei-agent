# 徐振伟智能体 — 智能推荐引擎 & 技法溯源 要件定义书

> 版本: v1.0 | 日期: 2026-06-17 | 作者: 徐振伟 & Claude

---

## 1. 业务要件

### 1.1 背景与课题

徐振伟智能体（以下简称 AAI）的核心交互流程是：

```
用户输入问题 → 系统推荐技法/配方 → 用户选择 → 执行分析 → 输出结果
```

**当前痛点：**

| # | 痛点 | 影响 |
|---|------|------|
| 1 | 无论问题简单还是复杂，始终推荐9条技法 | 简单问题被9张卡片淹没，用户决策疲劳 |
| 2 | 推荐卡片数量与问题复杂度不匹配 | "什么是酵素菌？"和"全面诊断农场"得到同样的关注度 |
| 3 | 导出报告中无法追溯使用了哪些技法 | 用户拿报告给第三方看时，无法证明分析方法的专业性 |
| 4 | 技法溯源信息丢失在流式传输中 | 前端接收SSE流后只显示内容，不保留技法元数据 |

### 1.2 目标

1. **动态推荐数量**: 根据问题复杂度，AI自动判定推荐1-8条技法（非必填满）
2. **技法溯源**: Word/PPT/Excel导出报告中自动标注"本次分析使用了以下技法：[038]行动计划生成法、[001]跨界特征联想法…"章节
3. **设计文档化**: 从要件→基本→详细的完整设计文档，便于后续维护和扩展

### 1.3 成功指标

| 指标 | 目标值 |
|------|--------|
| 简单问题推荐数 | 1-2条（波动范围 ±1条可接受） |
| 复杂问题推荐数 | 5-7条 |
| 分类准确率 | >85%（人工抽样校验） |
| 技法溯源完整率 | 100%（每次分析都带技法标注） |
| 推荐响应延迟 | <50ms（本地判断，不调API） |

## 2. 機能要件

### FR-01: 问题复杂度自动分类

**入力**: 用户输入文本（String）
**出力**: 复杂度等级（SIMPLE_FACTUAL / SINGLE_DOMAIN / COMPLEX_MULTI / COMPREHENSIVE）
**判定ロジック**: 关键词模式匹配 + 多领域交叉检测（纯本地，不调API）

### FR-02: 推荐数量动态截断

**入力**: 排序后的匹配技法列表 + 复杂度等级
**出力**: 截断后的技法列表（1-8条）
**判定ロジック**: 复杂度等级确定上限，置信度断崖确定截断点

### FR-03: 推荐响应携带复杂度元数据

**API**: `POST /api/recommend/quick` 返回新增 `complexity` 字段
```json
{
  "complexity": {
    "level": "SIMPLE_FACTUAL",
    "recommendCount": 2,
    "maxAllowed": 2
  }
}
```

### FR-04: 导出报告技法溯源

**対象フォーマット**: Word (.docx), Excel (.xlsx), PPT (.pptx)
**追加コンテンツ**: 报告末尾追加技法溯源章节：
```
## 📋 分析方法溯源
本次分析使用了以下徐振伟智能体技法：
- [038] 行动计划生成法 — 把想法变成待办清单
- [001] 跨界特征联想法 — 从动物身上找灵感
分析模式: 🧠 深度思考 | 生成时间: 2026-06-17 15:30
```

### FR-05: 前端技法元数据传递

前端在 `send()` 时保存 `currentTechLabel`，导出时从消息DOM中提取技法badge信息，随导出请求发送给后端。

## 3. 非機能要件

### NFR-01: 性能
- 复杂度分类必须在本地完成，不调外部API
- 推荐列表截断操作 O(n)，n ≤ 20

### NFR-02: 可维护性
- 复杂度关键词配置放在代码常量中（当前不复杂到需要YAML）
- 复杂度等级和推荐数量映射可独立调整

### NFR-03: 互換性
- 向后兼容：旧版前端不发送技法信息时，导出不含溯源章节
- API变更加字段不删字段

### NFR-04: 拡張性
- RecommendationDecisionEngine 预留 `aiReviewCount()` 接口（Phase 4 AI复核用）
- 复杂度等级 enum 可按需扩展

---

## 4. 制約事項

1. 第一版（Phase 1-2）不做AI复核调用，纯本地规则
2. 复杂度分类基于关键词匹配，不支持语义理解（AI模式留给Phase 4）
3. 技法溯源依赖前端正确传递技法信息，流式中断可能导致溯源不完整
